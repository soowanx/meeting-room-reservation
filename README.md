## 프로젝트 실행 방법 (Docker 기반)

프로젝트 루트에서 아래 명령어를 실행합니다.

```bash
git clone <repository-url>
cd meeting-room-reservation
docker compose up --build -d
```

실행 후 접속 주소:

- 사용자 화면: `http://localhost:8080`
- 관리자 로그인: `http://localhost:8080/admin/login`

기본 관리자 비밀번호:

- `admin1234`

컨테이너 종료:

```bash
docker compose down
```

## 설계 내용과 설계시 고민했던 내용

### 1. 전체 구조

이 프로젝트는 Spring Boot, Thymeleaf, PostgreSQL, Docker Compose 조합으로 구성했습니다.

- 사용자: 회의실 조회, 필터링, 예약 생성, 내 예약 조회/취소
- 관리자: 로그인 후 회의실 등록, 회의실 활성화/비활성화, 전체 예약 조회/취소

### 2. 도메인 설계

핵심 엔티티는 `Room`과 `Reservation` 두 개입니다.

- `Room`: 이름, 위치, 수용 인원, 설명, 공간유형, 지역, 활성화 여부
- `Reservation`: 회의실, 예약자 이름/이메일, 사용 목적, 시작/종료 시간, 상태, 생성 시간

### 3. 예약 중복 방지

가장 중요한 사항은 같은 회의실에 겹치는 시간 예약이 등록되지 않도록 하는 것이라고 판단했습니다.

중복 판단 조건은 아래와 같습니다.

```text
새 예약 시작 < 기존 예약 종료 && 새 예약 종료 > 기존 예약 시작
```

예약 생성 시 같은 회의실의 확정 예약과 시간이 겹치면 저장하지 않도록 처리했고, 동시에 요청이 들어오는 상황을 조금 더 안전하게 다루기 위해 회의실 조회 시 `pessimistic lock`도 적용했습니다. 또한 서버에서도 30분 단위 시간만 허용하도록 검증했습니다.

### 4. 사용자 탐색 구조

목록 화면에서는 아래 조건으로 회의실을 탐색할 수 있게 했습니다.

- 공간유형
- 인원 범위
- 지역

회의실 개수가 늘어났을 때도 사용자가 후보를 빠르게 줄일 수 있도록 필터 구조를 먼저 잡았고, 결과 목록은 카드 형태로 보여주도록 구성했습니다.

### 5. 페이지 스크롤과 페이지네이션 사이의 선택

회의실 목록은 무한 스크롤 대신 페이지네이션을 적용했습니다.

- 한 페이지당 6개 회의실

이렇게 정한 이유는 다음과 같습니다.

- 필터와 함께 사용할 때 상태 관리가 단순하다
- 한 화면에서 카드 밀도가 과하지 않다

### 6. 인증 범위에 대한 고민

처음에는 사용자 로그인과 관리자 로그인을 모두 둘 수도 있었지만, 인증보다 예약 기능에 중점을 두었습니다.

그래서 현재는:

- 일반 사용자: 로그인 없이 예약 가능
- 관리자: 세션 기반 로그인

으로 단순화했습니다.

### 7. 개인정보 노출 범위

공개 회의실 상세 화면의 예정된 예약 목록에는 예약자 이름과 이메일을 노출하지 않도록 했습니다. 공개 화면에서는 시간 정보만 보여주고, 상세한 개인정보는 관리자 화면이나 본인 예약 조회 화면에서만 확인할 수 있게 구분했습니다.

### 8. 트랜잭션 경계와 정합성

예약 생성은 조회와 저장이 분리되면 중간에 상태가 바뀔 수 있기 때문에, 하나의 트랜잭션 안에서 처리하도록 구성했습니다.

먼저 같은 회의실에 겹치는 예약이 있는지 확인하고, 문제가 없을 때만 저장하도록 했습니다. 또한 동시에 같은 시간대 예약 요청이 들어오는 상황을 조금 더 안전하게 다루기 위해 `pessimistic lock`을 적용했습니다.

이번 과제에서는 완전한 분산 환경까지 가정하지는 않았지만, 예약 기능에서 가장 중요한 정합성을 우선해 설계했습니다.
