# 회의실 예약 관리 웹 어플리케이션

사용자는 조건에 맞는 회의실을 찾고 예약할 수 있고, 관리자는 회의실과 전체 예약 상태를 관리할 수 있는 Spring Boot 기반 웹 애플리케이션입니다.

이 프로젝트는 과제 제출을 목표로 만들었기 때문에, 짧은 시간 안에 핵심 사용 흐름을 완성하면서도 설계 의도를 설명할 수 있는 구조에 집중했습니다.

## 주요 기능

- 회의실 목록 조회
- 공간유형 / 인원 / 지역 필터
- 페이지네이션 기반 회의실 탐색
- 회의실별 예약 생성
- 30분 단위 시간 슬롯 선택
- 중복 시간대 예약 방지
- 이메일 기반 내 예약 조회 및 취소
- 관리자 로그인
- 회의실 등록 및 활성화/비활성화
- 전체 예약 조회 및 관리자 취소

## 기술 스택

- Backend: Java 17, Spring Boot, Spring MVC, Spring Data JPA, Thymeleaf
- Database: PostgreSQL
- Infra: Docker, Docker Compose
- Session: 기본 `JSESSIONID` 기반 세션 인증

## 실행 방법

프로젝트 루트에서 아래 명령어를 실행합니다.

```bash
docker compose up --build
```

백그라운드 실행:

```bash
docker compose up --build -d
```

실행 후 접속 주소:

- 사용자 화면: http://localhost:8080
- 관리자 로그인: http://localhost:8080/admin/login

기본 관리자 비밀번호:

```text
admin1234
```

원하면 `docker-compose.yml`의 `APP_ADMIN_PASSWORD` 값을 변경해서 사용할 수 있습니다.

컨테이너 종료:

```bash
docker compose down
```

## Docker 기반 제출/실행 안내

GitHub 저장소를 전달받은 담당자는 보통 아래 순서로 실행하면 됩니다.

```bash
git clone <repository-url>
cd meeting-room-reservation
docker compose up --build -d
```

주의할 점:

- `docker-compose.yml`이 아니라 최신 Docker CLI 기준 `docker compose` 명령을 사용했습니다.
- 처음 실행은 이미지 빌드가 필요하므로 `--build`를 붙이는 편이 안전합니다.
- 이미지를 한 번 빌드한 뒤에는 `docker compose up -d`만으로도 재기동할 수 있습니다.

## 설계 내용

### 1. 사용자 흐름과 관리자 흐름 분리

이 시스템은 역할이 명확히 나뉩니다.

- 사용자는 회의실을 찾고 예약합니다.
- 관리자는 회의실 메타데이터와 예약 상태를 관리합니다.

과제 범위를 고려하면 일반 사용자 인증까지 붙이기보다, 예약 핵심 기능과 관리자 관리 기능을 분리하는 편이 더 중요하다고 판단했습니다.

### 2. 예약 모델링

핵심 엔티티는 `Room`과 `Reservation` 두 개입니다.

#### Room

- 이름
- 위치
- 수용 인원
- 설명
- 공간유형
- 지역
- 활성화 여부

#### Reservation

- 회의실
- 예약자 이름
- 예약자 이메일
- 사용 목적
- 시작 시간
- 종료 시간
- 예약 상태
- 생성 시간

구조를 단순하게 유지한 이유는 다음과 같습니다.

- 과제에서 설명 가능한 모델이어야 함
- 예약 도메인의 핵심인 시간 충돌 방지에 집중해야 함
- 불필요한 테이블 분리보다 기능 완성도를 우선해야 함

### 3. 예약 중복 방지

가장 중요한 요구사항을 회의실 중복 예약 방지로 보고 아래 조건으로 처리했습니다.

```text
새 예약 시작 < 기존 예약 종료 && 새 예약 종료 > 기존 예약 시작
```

예약 생성 시 같은 회의실의 확정 예약 중 시간이 겹치는 데이터가 있는지 조회하고, 겹치면 저장하지 않습니다.

추가로 예약 생성 시에는 회의실 행을 `pessimistic lock`으로 조회해서, 동시에 겹치는 요청이 들어와도 더 안전하게 처리하도록 했습니다.

또한 서버에서도 30분 단위 시간만 허용하도록 검증했습니다.

### 4. 검색/탐색 구조

목록 화면에서는 다음 필터를 제공합니다.

- 공간유형
- 인원 범위
- 지역

이 필터는 사용자가 빠르게 후보군을 줄일 수 있게 해주고, 예약 기능에 들어가기 전 탐색 경험을 만드는 데 목적이 있습니다.

### 5. 페이지네이션

회의실 목록은 페이지네이션을 적용했습니다.

- 한 페이지당 6개 회의실

이 값을 선택한 이유:

- 필터 영역이 좌측에 고정된 레이아웃이라 우측 카드가 너무 많으면 한 화면 집중도가 떨어짐
- 데스크톱 기준 2행 정도에서 정보량이 적당함
- 과제 검토 시 스크롤 길이가 과도하지 않아서 확인이 쉬움

## 설계하면서 고민했던 내용

이 섹션은 기술적 고민과 비기술적 고민을 함께 정리했습니다.

### 1. 인증을 어디까지 구현할지

처음에는 사용자 로그인과 관리자 로그인을 모두 넣는 방향도 생각할 수 있었습니다. 하지만 과제 시간 안에 핵심 기능을 완성하려면 인증보다 예약 생성, 중복 방지, Docker 실행 안정성이 더 중요하다고 판단했습니다.

그래서 현재는:

- 일반 사용자: 로그인 없이 예약 생성
- 관리자: 세션 기반 로그인

으로 단순화했습니다.

여기서 Spring Security는 일부러 넣지 않았습니다. 이유는 다음과 같습니다.

- 관리자 비밀번호 기반 세션 로그인 정도는 현재 구조로도 충분함
- 과제 범위에서 Security 설정까지 들어가면 코드량이 빠르게 증가함
- 제출용 프로젝트에서는 핵심 도메인 로직이 더 잘 보이는 편이 유리함

다만 실제 서비스로 확장한다면 다음 단계에서는 Spring Security를 도입하는 편이 맞습니다.

- 사용자/관리자 권한 분리
- 세션 만료 정책
- CSRF/인가 규칙
- 로그인 실패 처리

### 2. 프론트엔드를 별도 SPA로 분리할지

React/Vue로 프론트엔드를 분리하면 UI 자유도는 높아지지만, 이번 과제에서는 다음 비용이 컸습니다.

- 프로젝트 구조 복잡도 증가
- Docker 설정 증가
- 제출 직전 검증 포인트 증가

그래서 이번에는 Thymeleaf 기반으로 구성해 백엔드 중심 과제라는 인상을 유지하면서도 화면 완성도를 확보하는 방향을 택했습니다.

### 3. 페이지 스크롤 vs 페이지네이션

회의실 목록 탐색 방식은 두 가지를 고민했습니다.

#### 무한 스크롤 / 긴 목록

장점:

- 사용자가 자연스럽게 더 많은 회의실을 볼 수 있음
- 탐색형 서비스에 익숙한 UX

단점:

- 필터 조합이 바뀌면 스크롤 위치 관리가 번거로움
- 과제 검토 시 끝까지 내려봐야 해서 확인성이 떨어짐
- 서버/클라이언트 상태 관리가 늘어남

#### 페이지네이션

장점:

- 현재 결과 개수와 범위를 설명하기 쉬움
- 검토자 입장에서 재현과 확인이 쉬움
- 필터와 함께 동작할 때 상태 관리가 단순함

단점:

- 탐색 흐름이 조금 끊길 수 있음

이번 과제에서는 “서비스형 탐색 UX”보다 “명확하게 설명 가능한 구조와 검증 가능성”이 더 중요하다고 봐서 페이지네이션을 선택했습니다.

### 4. 개인정보 노출 범위

공개 회의실 상세 화면의 예정된 예약 목록에서 예약자 이름과 이메일을 보여주는 것은 좋지 않다고 판단했습니다.

그래서 현재는 공개 화면에서는 시간만 보여주고, 예약자 정보는 노출하지 않도록 수정했습니다. 개인정보는 최소 노출 원칙을 따르는 편이 더 자연스럽고, 실제 서비스 방향과도 가깝다고 봤습니다.

관리자 화면과 사용자 본인 조회 화면은 관리 목적과 소유자 확인 목적이 있으므로 상세 정보를 유지했습니다.

### 5. 더미 데이터 범위

단순히 회의실 3개만 있으면 필터와 페이지네이션의 의미가 약해집니다.

그래서 더미 데이터를 늘릴 때 다음을 고려했습니다.

- 지역이 여러 개로 퍼져 있어야 필터가 살아남
- 공간유형이 다양해야 체크박스 필터가 의미가 생김
- 인원이 다양해야 범위 슬라이더가 의미가 생김

즉, 더미 데이터는 단순한 장식이 아니라 기능 검증용 데이터로 설계했습니다.

### 6. Redis, Kafka, 모니터링 도입 여부

이번 과제에서는 일부러 넣지 않았습니다.

#### Redis를 쓰지 않은 이유

- 현재 트래픽 가정에서는 PostgreSQL만으로 충분함
- 캐시보다 예약 정합성이 더 중요함
- 설명 가능한 범위를 벗어나지 않기 위해서

#### Kafka를 쓰지 않은 이유

- 예약 생성/취소는 즉시성 있는 동기 처리에 가까움
- 이벤트 스트리밍이 핵심 요구사항이 아님

#### 모니터링을 붙이지 않은 이유

- 운영형 프로젝트가 아니라 과제 제출형 프로젝트임
- 대신 Docker 로그와 구조 설명으로 운영 관점을 최소한 보여주는 쪽이 더 적절하다고 판단함

## 비기술적 고민

### 1. 어디까지 만들고 어디서 멈출지

과제에서는 기술을 많이 넣는 것보다, 끝까지 동작하는 범위를 정확히 선택하는 것이 더 중요하다고 생각했습니다.

그래서 다음은 의도적으로 뺐습니다.

- 일반 사용자 회원가입/로그인
- 알림
- 반복 예약
- 캘린더 연동
- 복잡한 권한 체계

### 2. 과제 검토자의 입장에서 보기 쉬운가

이번 프로젝트는 “실제 서비스 100% 구현”보다 “짧은 시간 안에 판단 가능한 품질”이 중요합니다.

그래서 다음 기준을 계속 유지했습니다.

- Docker 한 번으로 실행될 것
- README만 읽어도 구조가 보일 것
- 핵심 기능이 눈에 띄게 동작할 것
